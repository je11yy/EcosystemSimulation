# EcosystemSimulation: техническая документация проекта

## 1. Назначение проекта

`EcosystemSimulation` - это многосервисный проект для моделирования экосистемы, в которой:

- есть граф территорий с ресурсами и температурными условиями;
- по территориям перемещаются агенты;
- поведение и характеристики агентов задаются геномами;
- геномы могут рекомбинироваться и мутировать при размножении;
- состояние симуляции можно собирать, запускать, ставить на паузу, шагать по тикам и останавливать;
- результаты симуляции доступны через API и визуализируются во frontend.

Проект разделен на 4 основные части:

1. `api_server` - основной backend, работающий с БД и frontend.
2. `simulation_core` - чистое ядро симуляции без привязки к FastAPI и БД.
3. `simulation_service` - runtime-сервис, который держит собранные симуляции в памяти и исполняет шаги.
4. `frontend` - пользовательский интерфейс.


## 2. Архитектура верхнего уровня

### 2.1 Поток данных

Упрощенно жизненный цикл симуляции выглядит так:

1. Пользователь создает симуляцию, территории, агентов и геномы через `frontend`.
2. `frontend` обращается в `api_server`.
3. `api_server` сохраняет данные в PostgreSQL.
4. Когда пользователь нажимает `Собрать` / `Шаг` / `Запустить`, `api_server`:
   - читает симуляцию из БД;
   - собирает runtime-payload;
   - отправляет его в `simulation_service`.
5. `simulation_service` через `simulation_core`:
   - строит `Engine`;
   - выполняет шаги;
   - хранит runtime в памяти.
6. `api_server` получает snapshot/result, синхронизирует БД и сохраняет логи шагов.
7. `frontend` получает обновленное состояние симуляции, метрики и логи.

### 2.2 Границы ответственности

- `api_server` отвечает за:
  - авторизацию;
  - CRUD-операции;
  - хранение постоянных данных;
  - сценарии и шаблонные геномы;
  - синхронизацию runtime-состояния обратно в БД.

- `simulation_core` отвечает за:
  - модели мира;
  - модели агентов и геномов;
  - политику выбора действий;
  - применение действий;
  - рождение, смерть, конфликты, охоту;
  - расчёт метрик и логов шага.

- `simulation_service` отвечает за:
  - хранение собранных симуляций в оперативной памяти;
  - команды `build`, `step`, `start`, `pause`, `stop`;
  - отдачу snapshot и накопленных runtime-результатов.

- `frontend` отвечает за:
  - пользовательские сценарии работы;
  - формы редактирования;
  - визуализацию графов территорий и геномов;
  - просмотр логов, последнего шага и графиков метрик.

## 5. Геномы и гены

## 5.1 Общая модель

Геном представлен как ориентированный взвешенный граф:

- вершины - гены;
- ребра - связи между генами;
- у каждого гена есть тип эффекта, порог активации и сила влияния.

В `simulation_core` ген не считается просто статической записью. Его активность вычисляется динамически на основе:

- состояния агента;
- температуры территории;
- плотности популяции на территории;
- удовлетворенности;
- других факторов окружения.

Поле `is_active` хранится только в runtime ядра симуляции и не сохраняется в БД.

## 5.2 Эффекты генов

Поддерживаемые `GeneEffectType`:

### Базовые характеристики

- `MAX_HP` - множитель максимального здоровья.
- `STRENGTH` - множитель силы.
- `DEFENSE` - множитель защиты.
- `METABOLISM` - влияние на энергетическую стоимость и обмен веществ.

### Базовое поведение

- `HUNGER_DRIVE` - склонность к поиску еды.
- `DISPERSAL_DRIVE` - склонность к перемещению.
- `SITE_FIDELITY` - склонность оставаться на текущей территории.
- `REPRODUCTION_DRIVE` - склонность к спариванию.

### Среда

- `HEAT_RESISTANCE` - устойчивость к жаре.
- `COLD_RESISTANCE` - устойчивость к холоду.

### Конфликты и охота

- `AGGRESSION_DRIVE` - склонность к конфликтам.
- `PREDATION_DRIVE` - склонность к охоте.
- `CARNIVORE_DIGESTION` - выгода от успешной охоты.
- `CANNIBAL_TOLERANCE` - терпимость к нападению на генетически похожих агентов.

### Социальность и эволюция

- `SOCIAL_TOLERANCE` - переносимость высокой плотности популяции.
- `MUTATION_RATE` - множитель вероятности мутаций потомства.

## 5.3 `threshold` и `weight`

### `threshold`

Порог активации гена. Интерпретируется в зависимости от эффекта:

- для `HUNGER_DRIVE` - порог голода;
- для `HEAT_RESISTANCE` - температура;
- для `SOCIAL_TOLERANCE` - число агентов на территории;
- для некоторых базовых физиологических генов порог может фактически не использоваться.

### `weight`

Сила гена:

- для статовых генов - множитель;
- для поведенческих генов - вес склонности;
- для `MUTATION_RATE` - множитель вероятностей мутации.

### `default_active`

Если `true`, ген работает независимо от того, достигается ли порог в текущем состоянии.


## 6. Шаблонные геномы

Шаблоны задаются в [definitions.py](/Users/lvlasada/university/EcosystemSimulation/api_server/app/services/scenario/definitions.py).

Сейчас в проекте есть:

1. `balanced_grazer` - базовый травоядный.
2. `heat_migrant` - термоустойчивый мигрант.
3. `cold_social` - холодоустойчивый социальный агент.
4. `predator` - хищник.
5. `mutable_pioneer` - агент с повышенной изменчивостью.

Назначение шаблонов:

- дать пользователю готовые геномы для экспериментов;
- использовать их в готовых сценариях;
- защитить базовые контрольные конфигурации от редактирования.

Шаблонный геном:

- виден в списке геномов;
- доступен для использования при создании агентов;
- недоступен для редактирования и удаления.

## 8. Симуляция: логика шага

Один тик `Engine.step()` в общем случае включает:

1. обновление активности генов у каждого живого агента;
2. построение наблюдения агента;
3. расчет фенотипа и трейтов;
4. выбор действия политикой;
5. применение действий;
6. разрешение конкуренции за еду и конфликтов;
7. разрешение охоты;
8. применение стоимости действий;
9. рождение потомства;
10. голод, урон, смерти;
11. регенерацию ресурсов территорий;
12. расчет метрик;
13. формирование `StepResult` и событий шага.

Дополнительно:

- при спаривании используется рекомбинация геномов;
- при рождении возможно применение мутаций;
- удовлетворенность пересчитывается как отдельная характеристика условий среды.


## 9. Сравнение, совместимость, рекомбинация и мутации геномов

## 9.1 Совместимость геномов

`GenomeCompatibilityCalculator` сравнивает два генома по трем направлениям:

- состав и параметры генов;
- структура ребер;
- топология графа.

Результат:

- `score` - мера совместимости;
- `distance` - мера различия;
- `matched_genes` - сопоставление гомологичных генов.

## 9.2 Рекомбинация

`GenomeRecombinator` строит геном потомка на основе матери и отца.

Текущая логика специально делает потомка ближе к материнскому графу:

- материнский геном используется как структурный шаблон;
- параметры совпавших генов смешиваются со смещением к материнским;
- уникальные отцовские гены и ребра наследуются осторожно;
- расстояние ребенка от материнского генома ограничивается.

## 9.3 Мутации

`GenomeMutator` применяет редкие изменения:

- изменение `threshold`;
- изменение `weight`;
- переключение `default_active`;
- дублирование генов;
- добавление новых генов;
- добавление/удаление/изменение ребер.

Текущая конфигурация делает мутации редкими, а удаление генов отключено, чтобы геномы не деградировали слишком быстро.


## 10. Runtime-управление симуляциями

У симуляции есть два уровня состояния:

- persistable state в БД;
- runtime state в памяти `simulation_service`.

Поддерживаются команды:

### `build`

Собирает симуляцию из БД в `Engine` и размещает её в памяти.

### `step`

Выполняет один тик симуляции и возвращает `StepResult` плюс snapshot.

### `start`

Запускает последовательное выполнение шагов в фоне.

### `pause`

Останавливает фоновый цикл, но runtime остается в памяти.

### `stop`

Удаляет runtime-симуляцию из памяти.

`api_server` синхронизирует runtime обратно в БД через `runtime_orchestrator` и `runtime_persister`.


## 11. Логи и метрики

## 11.1 Логи

`simulation_logs` хранят:

- решения агентов;
- результаты применения действий;
- смерти;
- рождения;
- конфликты;
- охоты;
- агрегированные события шага.

В UI:

- последние логи показываются в модалке;
- можно выгрузить расширенный лог по тикам.

## 11.2 Метрики

В метриках шага и их истории используются, в частности:

- живая популяция;
- средний голод;
- средняя удовлетворенность;
- количество съеденной еды;
- успешные/неуспешные охоты;
- рождения;
- смерти;
- распределение по территориям;
- причины смертей.


## 12. Отличие пользовательских, шаблонных и runtime-геномов

### Пользовательские геномы

- создаются через UI;
- принадлежат пользователю через `genome_user_relations`;
- отображаются в списке геномов;
- редактируемы.

### Шаблонные геномы

- создаются системой;
- помечены `is_template = true`;
- видны пользователю;
- не редактируются.

### Runtime-геномы потомства

- создаются во время симуляции;
- привязываются к агентам;
- доступны по прямой ссылке, если связаны с агентом пользователя;
- не попадают в пользовательский список геномов;
- доступны только в режиме просмотра.


## 13. Где лучше продолжать развитие проекта

На текущем этапе наиболее естественные точки расширения:

1. Усложнение регуляторики генов:
   - влияние знака и веса ребра на активацию downstream-генов;
   - каскадные эффекты внутри генома.

2. Балансировка симуляции:
   - цены действий;
   - пищевые циклы;
   - reproduction windows;
   - predator-prey balance.

3. Аналитика:
   - сравнение симуляций;
   - экспорт логов и метрик;
   - графики выживаемости и распределения геномов.

4. Улучшение frontend:
   - более развитая визуализация генетических графов;
   - сравнение генома агента с шаблоном;
   - diff между геномами родителей и потомка.

5. Уточнение БД-модели:
   - выделение отдельной таблицы runtime-derived genomes при необходимости;
   - хранение версий генома;
   - отдельная нормализация аналитических метрик.


## 14. Краткий технический итог

Проект уже построен как хорошая многослойная система:

- БД хранит постоянные сущности и историю;
- `api_server` управляет жизненным циклом данных;
- `simulation_core` содержит всю вычислительную логику;
- `simulation_service` отделяет runtime от основного API;
- `frontend` дает визуальный контроль над симуляцией, геномами и логами.

Это удобная архитектура для дальнейшего развития в сторону:

- исследовательского симулятора;
- учебной платформы;
- песочницы для анализа эволюционных сценариев.
